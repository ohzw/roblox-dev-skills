## レベル 1: LLM アーキテクチャの本質的制約

### L1-1: トークン逐次生成 vs 空間的同時整合性

> **課題**: LLM はトークンを逐次生成するが、3D ジオメトリはすべての座標が同時に整合することを要求する。パーツ B をパーツ A 基準で配置する際、A の寸法はアテンションで「記憶」されているだけで幾何学的に検証されない。

#### (A) コミュニティ・研究者の知見

**対策は存在する。複数のアプローチが実証されている。**

**1. 制約ソルバーとの併用（最も学術的に検証済み）**

LLM が絶対座標ではなく**空間的関係（制約）**を生成し、別の制約ソルバーが最終座標を解決するパターン。

- **Open-Universe Indoor Scene Generation** (Brown/MIT, 2024, arXiv:2403.09675): LLM がドメイン固有レイアウト言語でプログラムを生成 → 制約充足問題（CSP）仕様に変換 → 勾配ベースの最適化ソルバーが最終位置を決定。直接的な LLM レイアウトよりも高精度。
- **Holodeck** (2023, arXiv:2312.09067): GPT-4 がシーン記述 + オブジェクト配置の「関係性」を生成 → 制約ソルバーが衝突検出と空間最適化を処理。

**2. プログラム的な相対座標参照（業界標準）**

絶対座標のハードコードではなく、変数と相対式でコードを生成するパターン。

- **Adam CAD** (2025): `wall_thickness = 3; height = wall_thickness * 4` のようにパラメータ名で参照。パラメータ調整は AI を介さず正規表現で決定的に処理。
- **ShapeAssembly** (MIT/Stanford, 2020, arXiv:2009.08026): パーツをグローバル座標ではなく**他のパーツに対する相対位置**で定義するドメイン固有言語。

**3. 2 フェーズ生成（計画→実行）**

- **GALA3D** (2024, arXiv:2402.07207): フェーズ 1 で LLM が粗いレイアウト（バウンディングボックス）を生成 → フェーズ 2 で幾何学的制約に基づき精緻化。
- **Database-Driven 3D Level Generation** (2025, arXiv:2508.18533): LLM は座標を生成せず、事前検証済みコンポーネントの**選択と配置**のみを行う。

**4. 空間スクラッチパッド**

- **3D Space as a Scratchpad** (2026, arXiv:2601.14602): VLM の Chain-of-Thought を空間推論に適用。明示的な 3D ワークスペースで空間推論を外部化してから出力を生成。

#### (B) AI の見解

**最も即効性が高いのは「相対座標パターン」**である。LLM が以下のようなコードを生成するだけで、整合性問題の大部分を緩和できる:

```lua
local SEAT_SIZE = Vector3.new(4, 0.5, 4)
local SEAT_Y = 4
-- 脚は天板の座標から導出（絶対座標の再計算をしない）
local LEG_HEIGHT = SEAT_Y - SEAT_SIZE.Y / 2
local LEG_Y = LEG_HEIGHT / 2
```

これは特別なインフラを必要とせず、**プロンプトエンジニアリングだけで実現可能**。制約ソルバーの併用はより堅牢だが、実装コストが高い。

| 対策 | 実用性 | 実装コスト | 効果 |
|------|--------|-----------|------|
| 相対座標 + 変数参照 | 今すぐ可能 | 低（プロンプトのみ） | 高 |
| 2 フェーズ（計画→生成） | 今すぐ可能 | 低 | 中-高 |
| 実行フィードバックループ | 今すぐ可能 | 中 | 中 |
| 制約ソルバー併用 | 要実装 | 高 | 非常に高 |
| 空間スクラッチパッド (RL) | 研究段階 | 非常に高 | 将来的に高 |

### L1-2: 座標値のトークン分解による精度問題

> **課題**: LLM は数値を文字単位でトークン化する（`-4.75` = 5 トークン）。各桁に独立したエラー確率があり、3D 座標の精度を損なう。

#### (A) コミュニティ・研究者の知見

**対策は存在する。トレーニング時の根本解決と、デプロイ済みモデルへの実用的回避策の 2 層がある。**

**1. 根本的解決（トレーニング時）**

- **BitTokens** (2025, arXiv:2510.06824): 任意の数値を IEEE 754 バイナリ浮動小数点表現で**単一トークン**にエンコード。小規模モデルで基本算術がほぼ完璧に解けることを実証。ただし、Claude / GPT-4 のようなデプロイ済みモデルには適用不可。
- **TabDLM** (2026, arXiv:2602.22586): 「離散トークン化は精密な数値を歪める」ことを確認。連続拡散プロセスと特殊な数値トークン埋め込みのハイブリッドで対処。

**2. デプロイ済みモデルへの回避策**

| 戦略 | 仕組み | 効果 |
|------|--------|------|
| **整数グリッドシステム** | すべての座標を整数に制約（例: `Size = Vector3.new(4, 2, 6)`） | 極めて高。小数トークン化を完全に排除 |
| **名前付き定数 / 変数** | `local WALL_HEIGHT = 10` として名前で参照 | 高。再計算を減らし、文脈にセマンティクスを保持 |
| **丸め規約のプロンプト指定** | 「すべての座標は 0.5 の倍数であること」 | 中。トークン数は減るがドリフトの可能性 |
| **構造化出力 (JSON)** | スキーマ: `{"x": int, "y": int, "z": int}` を強制 | 高。フォーマットを制約しハルシネーションの余地を縮小 |
| **計算の外部化 (CoT)** | 「まず計算: base_y = floor_y + height/2 = 0 + 5 = 5」 | 高。算術を外部化し、エラーカスケードを削減 |

#### (B) AI の見解

**整数グリッドは最も費用対効果が高い単一の対策**である。Roblox のネイティブなスタッドシステムは本来整数ベースであり、すべての座標を整数に制約するだけで小数のトークン化問題を完全に排除できる。`-4.75`（5 トークン、エラーリスク高）→ `-5`（2 トークン、自明）。

ただし、微妙な位置調整（0.25 スタッドのオフセットなど）が必要な場面では整数制約が表現力を損なう。この場合は変数参照（`local OFFSET = 0.25`）と Chain-of-Thought 計算の併用が有効。

### L1-3: アテンション距離と参照精度の低下

> **課題**: コンテキスト長が増えるほど、初期に定義したパーツの寸法情報への注意が分散し、後続パーツの座標精度が低下する。

#### (A) コミュニティ・研究者の知見

**対策は存在する。「Lost in the Middle」問題として広く研究されている。**

- **Lost in the Middle** (2025, arXiv:2510.10276): LLM のパフォーマンスはコンテキストの**中間**にある情報で最も劣化し、先頭（primacy）と末尾（recency）は比較的保持される U 字型カーブを示す。
- **Hidden in the Haystack** (2025, arXiv:2505.18148): 小さな情報スニペット（例: 単一の座標値 `Size = Vector3.new(4, 2, 6)`）は大きなスニペットよりも検索が困難。
- **AdmTree** (NeurIPS 2025, arXiv:2512.04550): 階層的セマンティックツリーによるコンテキスト圧縮で位置バイアスを緩和。「gist トークン」でセグメントを要約。

**実用的対策:**

| 戦略 | 仕組み | 効果 |
|------|--------|------|
| **ジオメトリサマリヘッダー** | 各生成呼び出しの**先頭**に配置済みパーツのコンパクトな要約を挿入 | 極めて高（primacy 効果を活用） |
| **変数ベースの参照** | `local BODY_CENTER = Vector3.new(0, 5, 0)` → 後続パーツが `BODY_CENTER.Y + 3` で参照 | 極めて高（「アクティブ」コード文脈に保持） |
| **チャンク分割生成** | 3-5 パーツごとに生成、要約してから次のチャンクへ | 高（中間コンテキスト埋没を防止） |
| **明示的な再掲** | 「Body (Part 1) の Size=(4,8,4), CFrame=(0,4,0)。次に Part 6 を追加...」 | 中（冗長だが効果的） |

#### (B) AI の見解

**ジオメトリサマリヘッダーとチャンク分割生成の組み合わせ**が最も実用的。具体的には:

```lua
--[[
  GEOMETRY MANIFEST (自動挿入)
  floor:  pos=(0,0,0),    size=(20,1,20)
  wall_N: pos=(0,5,-10),  size=(20,10,1)
  wall_S: pos=(0,5,10),   size=(20,10,1)
  -- ここまでで 3 パーツ配置済み
]]

-- 次のチャンク: 家具を配置
local TABLE_Y = 0 + 1/2 + 3/2  -- floor top + table half-height
```

これは L1-1 の変数参照パターン、L1-2 の整数グリッド、L1-3 のサマリヘッダーを**同時に解決する統合的な戦略**。3 つの課題は独立ではなく、1 つのコード生成パターンで同時に対処可能。

### L1-4: 視覚フィードバック無しの盲目的生成

> **課題**: AI はレンダリング結果を見ずにコードを生成する。CSG のブーリアン演算で生じたアーティファクトや、プロポーションの逸脱を検知できない。

#### (A) コミュニティ・研究者の知見

**対策は存在する。ただし、ターンキーなソリューションは 2026 年 2 月時点で存在しない。**

**1. VLM フィードバックループ（学術的に最も検証済み）**

- **CADCodeVerify** (Georgia Tech, 2024, arXiv:2410.05340): LLM が CAD コードを生成 → レンダリング → VLM (GPT-4V) が構造化検証質問を生成（「中央に穴はあるか？」）→ 回答をフィードバック → 修正。GPT-4 で Point Cloud 距離 7.3% 改善。
  - 重要な知見: VLM は自由形式の批評ではなく**構造化された検証質問**を生成するほうが信頼性が高い。
- **Iterative Refinement for Compositional Image Generation** (CMU/Meta, 2026, arXiv:2601.15286): 同原理を 2D に適用。T2I-CompBench 3D-Spatial カテゴリで 13.8% 改善。

**2. 業界ツールの現状**

| ツール | 視覚フィードバック | 状態 |
|--------|-------------------|------|
| Adam CAD | 人間のみ（VLM は計画中） | OpenSCAD WASM → Three.js でレンダリング |
| ChatToSTL | なし | 開発者自身が「AI can't see the render, so no self-correction yet」と明言 |
| Roblox MCP | なし | `run_code` はテキスト出力のみ。スクリーンショット機能なし |

**3. プログラム的な幾何学検証（非視覚的代替）**

視覚フィードバックが不可能な環境での代替手段:

| 検証タイプ | Roblox での実装 | 検出可能な問題 |
|-----------|----------------|--------------|
| バウンディングボックス検証 | `model:GetBoundingBox()` | 全体サイズの逸脱 |
| 交差検出 | `GetTouchingParts()` / `FindPartsInRegion3()` | 意図しないパーツの重なり |
| パーツ数検証 | `#model:GetDescendants()` | パーツ数の異常 |
| 空間関係アサート | CFrame 演算 | パーツ A がパーツ B の上にあるか等 |
| CSG 結果検証 | pcall で CSG エラーを捕捉 | ブーリアン演算の失敗 |

#### (B) AI の見解

**短期的に最も実用的なのはプログラム的検証**。Roblox MCP 環境では、コード実行後に検証スクリプトを自動で走らせ、結果をテキストでフィードバックする方式が現実的:

```lua
-- 生成後の自動検証
local model = workspace:FindFirstChild("GeneratedModel")
local cf, size = model:GetBoundingBox()

-- サイズの妥当性チェック
assert(size.X > 0 and size.Y > 0 and size.Z > 0, "Empty geometry")
assert(size.X < 100 and size.Y < 100 and size.Z < 100, "Unreasonably large")

-- パーツ間の空間関係チェック
for _, part in model:GetDescendants() do
    if part:IsA("BasePart") then
        assert(part.Position.Y >= 0, part.Name .. " is below ground")
    end
end

print("Validation passed: " .. #model:GetDescendants() .. " parts, "
    .. "size=" .. tostring(size))
```

視覚フィードバックは中長期的な理想解だが、現時点では:
- OS レベルのスクリーンショット → VLM 評価は**可能だが高コスト**（1 反復あたり 15-30 秒、$0.01-0.03）
- Roblox Studio プラグインによる `EditableImage` → HTTP 転送は**可能だが高い実装コスト**
- プログラム的検証は**視覚的な「見た目」の問題を検出できない**（セマンティックエラー）という限界がある

| 対策 | 実用性 | 検出範囲 | コスト |
|------|--------|---------|------|
| プログラム的検証 | 今すぐ可能 | 構造的エラー | 低 |
| OS スクリーンショット + VLM | 今すぐ可能 | 構造的 + 視覚的 | 中 |
| Studio プラグイン + VLM | 要開発 | 最も広い | 高 |
| CADCodeVerify 型パイプライン | 要実装 | 構造的 + 視覚的 | 高 |

## レベル 2: CSG 表現手法の構造的限界

### L2-1: 有機形状・曲面の表現不可能性

> **課題**: 人体、動物、植物などの有機的な形状は曲率が連続的に変化し、CSG プリミティブでの近似には膨大なパーツ数が必要。

#### (A) コミュニティ・研究者の知見

**純粋な CSG での根本解決はない。ただし、実用的なワークアラウンドが複数存在する。**

**1. hull() チェーン（OpenSCAD、最も効果的な CSG 内手法）**

凸包を計算する `hull()` を連鎖させることで、非凸の有機形状を近似できる:

```scad
// 滑らかな「blob」ボディ — ブーリアン演算不要
hull() {
  translate([0, 0, 0])  sphere(r=10);
  translate([20, 0, 5]) sphere(r=8);
  translate([40, 0, 0]) sphere(r=6);
}
```

Roblox には `hull()` の直接的な等価物がないが、重なり合う Sphere/Wedge + Union で近似可能。

**2. ローポリ/スタイライズド美学の採用（最も実用的）**

コミュニティのコンセンサスは「ブロック美学と戦うと、10 倍のパーツで 20% の視覚改善にしかならない」。

- Minecraft スタイルの完全ボクセル化
- フラットシェーディングのローポリスタイル
- チビ / SD プロポーション（誇張された簡略化された解剖学）

**3. ハイブリッドアプローチ（構造は CSG、有機部分はメッシュ）**

プロの Roblox ゲームスタジオの標準的なワークフロー:
- CSG (UnionOperation) → 構造的・機械的パーツ（壁、フレーム、プラットフォーム）
- MeshPart (Blender → OBJ) → 有機的要素（キャラクター、木、岩、動物）

**4. 回転体 / 掃引体（OpenSCAD）**

花瓶、ボトル、きのこ、木の幹など対称性のある形状は 2D プロファイルを回転:

```scad
rotate_extrude($fn=64)
  polygon(points=[[0,0],[5,0],[8,10],[6,20],[3,30],[0,35]]);
```

Roblox では SpecialMesh + カスタム OBJ、または MeshPart として import。

#### (B) AI の見解

**この課題にはパラダイムシフトが必要**であり、CSG の枠内で解決しようとすること自体が誤り。AI の観点からは:

1. **スタイルの選択が最も重要な意思決定**。「リアルな犬」を CSG で作ろうとするのは根本的に間違い。「ブロックスタイルの犬」「ミニマルな犬」に目標を切り替えるべき。
2. **AI が生成対象を CSG 向きか判定する**ステップの導入。「この要求は CSG で実現可能か？」を評価し、不可能なら代替手段（MeshPart import の提案）へ分岐する。
3. 制約下での創造性が鍵。少ないプリミティブでの表現力最大化は、AI にとっても人間にとっても「制約付きデザイン」の問題。

| 対策 | CSG 内で完結 | 品質 | 実用性 |
|------|-------------|------|--------|
| hull() チェーン | はい（OpenSCAD） | 中 | Roblox では限定的 |
| ローポリスタイル採用 | はい | 中-高（美学次第） | 高 |
| ハイブリッド（CSG + Mesh） | いいえ | 高 | 高 |
| 回転体 / 掃引体 | いいえ（要メッシュ変換） | 高 | 中 |
| AI による CSG 適合性判定 | — | — | 高（メタ対策） |

### L2-2: フィレット / ロフト / ブレンドの非対応

> **課題**: CSG のブーリアン境界は常にハードエッジを生成する。滑らかな角の丸みや接線連続な遷移は原理的に表現できない。

#### (A) コミュニティ・研究者の知見

**CSG 単体での根本解決はない。ただし、外部ツールでの解決策が成熟している。**

**1. Minkowski 和による擬似フィレット（OpenSCAD）**

形状と小さな球の Minkowski 和でエッジを丸める:

```scad
minkowski() {
  cube([20, 10, 5]);
  sphere(r=1, $fn=32);  // 全エッジに 1mm のフィレット
}
```

- 利点: 任意の形状に適用可能、数学的に正確な丸み
- 欠点: O(n^2) の計算コスト、オブジェクトサイズが球の半径分膨張
- Roblox: 等価物なし

**2. build123d / CadQuery（CSG 後のパラダイム）**

OpenCASCADE をラップした Python ライブラリで、ネイティブなフィレット/ロフトを提供:

```python
with BuildPart() as part:
    Box(20, 10, 5)
    fillet(part.edges(), radius=1)   # 全エッジに NURBS フィレット
```

ロフト（異なる断面間の滑らかな遷移）も可能:

```python
with BuildPart() as lofted:
    with BuildSketch(Plane.XY) as s1:
        Circle(10)
    with BuildSketch(Plane.XY.offset(20)) as s2:
        Rectangle(15, 8)
    loft()  # 円から長方形への滑らかな遷移
```

STL/OBJ にエクスポートし、Roblox の MeshPart として import 可能。

**3. SDF / 暗黙的サーフェス（学術・先端）**

libfive や ImplicitCAD は Signed Distance Function でジオメトリを表現。形状間の滑らかなブレンドが自明:

```
blend(sphere, box, smoothness=0.3)
```

パーツ数のオーバーヘッドなしで有機的形状を生成。メッシュにエクスポートして Roblox で使用。

**4. 円柱フィレットトリック（OpenSCAD の古典的ワークアラウンド）**

2 つの面の交線に円柱を配置してフィレットを模擬:

```scad
translate([wall_thickness, 0, fillet_r])
  rotate([90,0,0])
    cylinder(h=depth, r=fillet_r);
```

手間がかかりエラーが起きやすいが、単純なケースでは機能する。

#### (B) AI の見解

**この制約は CSG の定義に内在するものであり、回避不可能**。対策の本質は「CSG の外に出る」か「制約を受け入れる」の二択。

AI の実用的アプローチ:
1. **ハードエッジを意匠として活用**。多くのゲームスタイルでは鋭いエッジが許容される（Roblox のデフォルト美学はブロック的）
2. **SmoothPlastic マテリアル + ライティング**で視覚的な角の印象を和らげる（形状は変えずに知覚を変える）
3. **フィレットが必要な場合は外部パイプラインを推奨**する判断を AI 自身が行う

### L2-3: パーツ数増加によるパフォーマンス劣化

> **課題**: プリミティブの数が増えるとパフォーマンスが低下する。AI は最適化を考慮せず生成する傾向がある。

#### (A) コミュニティ・研究者の知見

**対策は豊富に存在する。Roblox コミュニティで最も成熟した知見が蓄積されている領域。**

**1. CollisionFidelity の最適化（Roblox 公式ガイダンス）**

| 値 | コスト | 用途 |
|----|--------|------|
| `Box` | 最低 | 小型/非インタラクティブなオブジェクト |
| `Hull` | 中 | 中型オブジェクト |
| `PreciseConvexDecomposition` | 最高 | 複雑な衝突が必要な場合のみ |

非衝突オブジェクトではコリジョン自体を無効化。

**2. 静的ジオメトリの事前 Union**

Studio 内で事前にすべての CSG 操作を実行。ランタイムでの `UnionAsync()` は避ける。

**3. パーツマージヒューリスティック**

同じマテリアル・色で隣接する静的パーツは 1 つの PartOperation に Union（ドローコール削減）。

**4. LOD + ストリーミング**

`StreamingEnabled` + `ModelLOD` で遠方のジオメトリを自動的に簡略化。

**5. 定量的な目安（コミュニティで確立）**

| パーツ数 | 対応 |
|---------|------|
| < 2,000 | 懸念なし |
| 2,000 - 5,000 | CollisionFidelity を最適化 |
| > 5,000 | 積極的な LOD + ストリーミング必須 |

#### (B) AI の見解

**AI 側でできる最も効果的な対策は「パーツ予算」の導入**。生成開始時にターゲットパーツ数を設定し、それを超えないように生成する。

```
プロンプト設計:
"このオブジェクトを最大 30 パーツで構築してください"
```

加えて、AI は生成後に以下の最適化パスを実行できる:
1. **同マテリアル・隣接パーツの自動 Union**
2. **コリジョンが不要なパーツの `CanCollide = false` + `CollisionFidelity = Box` 設定**
3. **全静的パーツの `Anchored = true`**（物理シミュレーションのコスト排除）

| 対策 | 実装者 | 効果 |
|------|--------|------|
| パーツ予算の事前設定 | AI（プロンプト） | 高 |
| 生成後の自動 Union | AI（コード） | 高 |
| CollisionFidelity 自動設定 | AI（コード） | 中 |
| StreamingEnabled | 開発者（設定） | 高（大規模シーン） |

### L2-4: ブーリアン演算の精度問題 (epsilon / thin plane)

> **課題**: 2 つのパーツの面が共面のとき、CSG 演算結果が不定になる。薄い透明な平面が残留するアーティファクトが発生する。

#### (A) コミュニティ・研究者の知見

**対策は存在する。実用的ワークアラウンドからアルゴリズム的解決まで複数層がある。**

**1. epsilon オフセット（業界標準のワークアラウンド）**

Subtract するパーツを僅かに大きくして共面を回避:

```lua
local EPSILON = 0.01  -- studs
subtractPart.Size = subtractPart.Size + Vector3.new(EPSILON, EPSILON, EPSILON)
```

OpenSCAD コミュニティでは標準的なプラクティスとして確立:

```scad
epsilon = 0.01;
difference() {
  cube([10, 10, 10]);
  translate([-epsilon/2, -epsilon/2, -epsilon/2])
    cube([5 + epsilon, 5 + epsilon, 5 + epsilon]);
}
```

**2. Manifold ライブラリ（アルゴリズム的解決、最先端）**

OpenSCAD、Blender、Godot が採用する CSG ライブラリ ([github.com/elalish/manifold](https://github.com/elalish/manifold)):
- **Symbolic perturbation** を使用: ジオメトリが浮動小数点で完全に一致する場合、サーフェス法線方向に摂動を加える
- Union: 外側に摂動 → 接触する立方体がクリーンにマージ
- Difference: 内側に摂動 → 等高のカットが貫通穴を生成
- **出力のマニフォールド性を保証**する初のアルゴリズム

**3. 整数演算ベースの CSG（無条件ロバスト）**

- **Octree-Embedded BSPs** (2021, arXiv:2103.02486): 整数演算で無条件にロバストな操作。カスタム固定精度演算 + 効率的な平面切断アルゴリズム。

**4. 正確な述語 + Weiler モデル**

- Bruno Levy (2024, arXiv:2405.12949): 算術拡張を用いた正確な交点計算。Constrained Delaunay Triangulation + symbolic perturbation。

#### (B) AI の見解

**AI コード生成に直接組み込める対策は epsilon オフセットの自動適用**。すべての Subtract 操作で自動的に EPSILON を追加するルールを SKILL に組み込むことで、最も頻度の高いアーティファクトを予防できる:

```lua
-- AI が Subtract を生成する際の標準パターン
local EPSILON = 0.01
local function safeSubtract(mainPart, subtractPart)
    subtractPart.Size = subtractPart.Size + Vector3.new(EPSILON, EPSILON, EPSILON)
    return game:GetService("GeometryService"):SubtractAsync(mainPart, {subtractPart})
end
```

Roblox の CSG エンジンは Manifold ではなく独自実装のため、エンジン側の改善を待つことはできない。プロンプトレベルでの epsilon 適用が最も確実。

| 対策 | 実装レベル | 効果 | 適用可能性 |
|------|-----------|------|-----------|
| epsilon オフセット自動適用 | AI コード | 高 | 今すぐ可能 |
| 共面回避の設計ルール | AI プロンプト | 高 | 今すぐ可能 |
| Manifold ライブラリ | プラットフォーム | 非常に高 | Roblox には適用不可 |
| 整数演算 CSG | プラットフォーム | 非常に高 | Roblox には適用不可 |

### L2-5: CSG 分解の非一意性 (NP 困難)

> **課題**: 同じ形状を表す CSG ツリーは無数に存在する。AI は一貫した分解を選択できず、NP 困難な最適化問題に直面する。

#### (A) コミュニティ・研究者の知見

**根本解決は NP 困難のため存在しないが、実用的なヒューリスティックとニューラルアプローチが存在する。**

**1. CSG ツリー複雑性の学術的裏付け**

- **Friedrich & Fayolle (2021, arXiv:2103.06139)**: CSG ツリー抽出問題の探索空間がプリミティブ数に対して**指数的**であることを正式に証明。

**2. ヒューリスティックによる最適化**

- **CSG ツリー最適化パイプライン** (Friedrich et al., 2020, arXiv:2008.03674):
  - ステージ 1: 冗長性除去 + 分解（複雑性を削減）
  - ステージ 2: メタヒューリスティック最適化（遺伝的アルゴリズム、焼きなまし法）
  - **「編集性」メトリクス**を導入 — 最適なツリーは目的関数に依存する（操作数最小 vs 人間が編集しやすい vs 評価速度最速）

**3. 実用的なヒューリスティック**

| アプローチ | 仕組み | 特性 |
|-----------|--------|------|
| 貪欲なボリューム分解 | バウンディングボリュームを再帰的に分割 | 高速、準最適 |
| BSP-Net アプローチ | Binary Space Partitioning で正準分解 | 決定的、常にバイナリツリー |
| Union-of-Intersections 正規形 | DNF の幾何学版。任意の CSG を `Union(Intersection(...))` に | 正準形だがサイズが指数爆発しうる |
| 凸分解優先 | 形状を凸部品に分解 → 各部品を半空間の交差で表現 | 凸形状に強い |

**4. ニューラルアプローチ**

- **D^2CSG** (Yu et al., 2023, arXiv:2301.11497): 2 つの相補的ブランチ（Cover + Residual）+ Dropout でコンパクトなツリーを学習。全体形状を Union-of-Intersections でモデル化し、補集合（Subtract 領域）を明示的にモデル化。

#### (B) AI の見解

**NP 困難を解くのではなく、「十分良い」分解に到達するためのルールベースの指針**が実用的。AI がコード生成時に従うべき優先順序:

```
CSG ツリー構築の優先順序:
1. 支配的ボリュームを特定（最大の凸領域）
2. 穴/切り欠きの Subtract 操作を追加
3. 突起の Add 操作を追加
4. 同マテリアルの隣接プリミティブを Union 前にマージ
5. 浅いツリー（ネストレベルが少ない）を深いツリーより優先
```

重要な洞察: **3 つの課題（L2-3, L2-4, L2-5）は連動している**:
- パーツ数が少ない → ブーリアン演算が少ない → epsilon 問題の機会が減る
- 浅い CSG ツリー → 中間サーフェスが少ない → 共面アーティファクトの確率が減る
- 正準的なツリー形式 → アーティファクトの位置が予測可能 → epsilon 管理が容易

| 対策 | 実用性 | 最適性の保証 |
|------|--------|-------------|
| ルールベースの優先順序 | 今すぐ可能 | なし（ヒューリスティック） |
| 浅いツリー優先 | 今すぐ可能 | 部分的 |
| D^2CSG 型のニューラル分解 | 研究段階 | 学習データに依存 |
| メタヒューリスティック最適化 | 要実装 | 準最適 |

## レベル 3: プラットフォーム固有の技術的問題

### L3: Roblox / OpenSCAD 固有の問題

> **課題**: (a) 浮動小数点精度（原点から遠い位置でのジオメトリ崩壊）、(b) MeshPart と CSG の非互換性、(c) コリジョンフィデリティの不一致。

#### (A) コミュニティ・研究者の知見

**各問題に確立されたワークアラウンドが存在する。**

**(a) 浮動小数点精度**

IEEE 754 単精度浮動小数点は約 10,000 スタッドを超えると sub-stud の精度を失う。

| ワークアラウンド | 内容 |
|----------------|------|
| **原点付近で構築してから移動** | すべての CSG 操作を `(0,0,0)` 付近で実行し、完成後に PartOperation を最終位置に移動 |
| **StreamingEnabled + 原点シフト** | 大規模ワールドではストリーミングと原点シフトを組み合わせ |
| **ゲームプレイ重要ジオメトリに CSG を使わない** | 透明な衝突用 Part を MeshPart の上に重ねる |

**(b) MeshPart CSG 非互換性**

GeometryService の公式ドキュメントが明確に記載: 「`Part` と `PartOperation` のみサポート。`Terrain` や `MeshPart` はサポートされない。」

| ワークアラウンド | 内容 |
|----------------|------|
| **レイヤードコリジョン** | MeshPart を `CanCollide = false` にし、透明な Part で衝突形状を近似 |
| **外部で CSG 実行** | Blender でブーリアン演算を実行し、結果を MeshPart として import |
| **CSG 対象は Part のみ** | MeshPart は純粋に装飾要素として使用 |

**(c) コリジョンフィデリティの不一致**

| CollisionFidelity | 振る舞い | 推奨用途 |
|-------------------|---------|---------|
| `Default` | ボクセルベース凸包分解 | 高速だが穴/凹面で不正確 |
| `Hull` | 凸包 | 中型オブジェクト |
| `Box` | バウンディングボックスのみ | 最速、装飾オブジェクト |
| `PreciseConvexDecomposition` | HACD アルゴリズム | 最も正確、最もコストが高い |

#### (B) AI の見解

**これらは AI が「知っていれば回避できる」タイプの問題であり、SKILL ドキュメントに組み込むべき知識**。

具体的なルール:
1. すべての CSG 操作は `CFrame.new(0, 0, 0)` 付近で実行し、完成後に最終位置に `Model:PivotTo()` する
2. MeshPart に対して CSG 操作を試みない（エラーが発生する前に回避）
3. 装飾的な Union には `CollisionFidelity = Enum.CollisionFidelity.Box` を自動設定
4. ゲームプレイに影響する複雑な衝突形状が必要な場合は、透明な Part プロキシの使用を推奨

これらはルールとして明文化するだけで効果がある。AI がこの知識を持っていれば、そもそも問題に遭遇しない。

## 追加課題: 自然言語と幾何学の意味論的ギャップ

> **課題**: 「椅子を作って」は寸法、比率、正確な配置を指定しない。自然言語は本質的に曖昧であり、3D 座標は精度を要求する。

#### (A) コミュニティ・研究者の知見

**対策は存在する。業界は「中間表現」パターンに収束しつつある。**

**1. パラメトリックコードを中間表現として使用（業界標準）**

Adam CAD と ChatToSTL が採用するパターン:

```
自然言語 → LLM → パラメトリックコード（OpenSCAD / Luau） → ジオメトリ
```

パラメトリックコードが NL と座標の間の**橋渡し**になる。コードは意図（`seat_height`）と値（`45`）の両方を保持する。

**2. ドメイン固有言語 (DSL)**

- **ShapeAssembly** (MIT/Stanford, 2020, arXiv:2009.08026): キュボイドプロキシ + 階層的アタッチメント + 対称性オペレータ。パーツを他のパーツに対する相対位置で宣言。Roblox の Part ベースのビルドは本質的に ShapeAssembly と同型。
- **LayoutGPT** (Stanford, 2023, arXiv:2305.15393): CSS ライクな言語で空間レイアウトを生成。直接の text-to-image より空間正確性が 20-40% 向上。

**3. テンプレート / ライブラリベースのアプローチ**

「椅子」→ デフォルトパラメータ付きのパラメトリックテンプレートにマッピング → ユーザーがスライダーで調整 or 「脚を短くして」で修正。

BOSL2 ライブラリにはプリビルトのパラメトリック家具プリミティブがある。

**4. 仕様の例示（参照画像）**

Adam CAD: 参照画像をアップロード → Claude がプロポーションを分析 → マッチする OpenSCAD を生成。曖昧な用語（「モダン」「素朴」）を視覚的参照で接地。

**5. 即時生成 + 反復（多ターン明確化より高 UX）**

研究は、ユーザーが事前の Q&A よりも**即時生成 + 修正**を好むことを示す。Adam のアプローチ: デフォルトで生成 → パラメータを公開 → ユーザーが微調整。

#### (B) AI の見解

**AI がこのギャップに対処するために内部的に行うべきことは「暗黙の仕様を明示化する推論ステップ」**:

```
入力: 「椅子を作って」

AI の内部推論（CoT）:
- カテゴリ: 家具 → 椅子
- デフォルト仕様:
  - 座面高: 4 studs (Roblox スケール)
  - 座面サイズ: 4x4 studs
  - 脚: 4 本、各 0.5x0.5 studs
  - 背もたれ: 高さ 4 studs
- スタイル: Roblox 標準（ブロック的）
- パーツ予算: 10 パーツ以内
```

この推論ステップにより、曖昧なプロンプトが具体的な設計仕様に変換される。重要なのは**ユーザーに質問するのではなく、合理的なデフォルトで即座に生成し、修正を受け付ける**こと。

| 対策 | 実用性 | ユーザー体験 |
|------|--------|-------------|
| パラメトリックコード中間表現 | 今すぐ可能 | 高 |
| 暗黙仕様の内部推論 | 今すぐ可能（CoT） | 高 |
| テンプレートライブラリ | 要構築 | 非常に高 |
| 参照画像による接地 | VLM 必要 | 高 |
| 即時生成 + 反復修正 | 今すぐ可能 | 最も高い |

## 総合マトリクス

すべての課題と対策の対応関係を俯瞰する。

| # | 課題 | 対策の有無 | 最も実用的な対策 | 実装コスト | 根本解決か |
|---|------|-----------|----------------|-----------|-----------|
| L1-1 | トークン逐次生成 vs 空間整合性 | あり | 変数 + 相対座標パターン | 低 | 緩和 |
| L1-2 | 座標トークン化精度 | あり | 整数グリッドシステム | 低 | 排除 |
| L1-3 | アテンション距離劣化 | あり | ジオメトリサマリヘッダー + チャンク分割 | 低 | 緩和 |
| L1-4 | 視覚フィードバック欠如 | あり（部分的） | プログラム的検証 → OS スクリーンショット + VLM | 中-高 | 緩和 |
| L2-1 | 有機形状の表現不可能性 | なし（CSG 内） | スタイル変更 or ハイブリッド | 低-中 | パラダイムシフト |
| L2-2 | フィレット/ロフト非対応 | なし（CSG 内） | 外部ツール or 受容 | 低-高 | パラダイムシフト |
| L2-3 | パーツ数パフォーマンス | あり | パーツ予算 + 自動 Union + CollisionFidelity | 低 | 解決 |
| L2-4 | ブーリアン精度問題 | あり | epsilon オフセット自動適用 | 低 | 緩和 |
| L2-5 | CSG 分解の非一意性 | 部分的 | ルールベース優先順序 | 低 | ヒューリスティック |
| L3 | プラットフォーム固有問題 | あり | SKILL ルールとして明文化 | 低 | 回避 |
| Extra | NL-幾何学ギャップ | あり | 暗黙仕様の内部推論 + 即時生成反復 | 低 | 緩和 |

### 対策の相互作用

重要な発見として、**複数の課題を同時に解決するパターン**が存在する:

```
「名前付き変数 + 整数グリッド + ジオメトリサマリヘッダー」パターン
  → L1-1（空間整合性）+ L1-2（座標精度）+ L1-3（アテンション距離）を同時に解決

「パーツ予算 + 浅い CSG ツリー + epsilon 自動適用」パターン
  → L2-3（パフォーマンス）+ L2-4（ブーリアン精度）+ L2-5（分解の一貫性）を同時に緩和
```
